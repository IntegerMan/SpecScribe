## What Went Well

- **The "one seam" discipline held for six stories straight.** Every story routed new behavior through exactly one existing seam (`StatusStyles` for color/icon, `ModuleContext`/`SiteNav` for filename classification, one `Icons` library, one Markdig renderer-wrap pattern mirroring `MermaidCodeBlockRenderer`). No story forked a parallel palette or a second classifier.
- **"Companion of a primary doc" became a reusable pattern.** Story 2.2 invented it for spec companions (resolve-or-omit cross-links); Story 2.4 reused it verbatim for PRD/review-rubric (generate the page, suppress the card, link from the primary) — genuine cross-story pattern reuse.
- **Graceful degradation was load-bearing, not decorative.** Story 2.3's `SprintStatusParser` had to survive a genuinely invalid `story_location: {project-root}/…` line in this repo's own production yaml, producing the durable memory note `sprint-status-yaml-not-valid-yaml`.
- **Story 2.5's icon library landed as a true single source of truth on the first attempt** — 18 curated concept keys, one status-glyph seam, 11 badge call sites composed through one shared helper; review only found polish-level issues (duplicate glyphs, a stale doc comment), nothing structural.
- **Story 2.6 fixed a real, live bug in the project's own output** — the `epics.md:52` FR15–FR19 comment was silently vanishing from every generated page before this story, despite being small in scope.

## Challenges

- **Story 2.3 scope crept through eight unplanned "polish" passes.** What shipped as "a sprint page plus a dashboard widget" grew into retrospective pages, an action-items page, and a full Kanban board redesign — none of it in the original two ACs. This tangled with Story 2.5's work closely enough that 2.5's own review had to formally disclose it had accidentally carried a Story 2.3 redesign in its commits (`BmadCommands.cs`, `SprintTemplaterTests.cs`), because the two were interleaved in the working tree.
- **The story file stopped describing reality.** By the time Story 2.3 shipped, its own "Testing Requirements" section covered a fraction of what was actually built and tested — a real risk for anyone trusting the story file as a scope-of-record.
- **A stale generation command kept reappearing.** Three separate Epic 2 stories (2.1, 2.2, 2.4) independently repeated the known-wrong `--output docs/live` command in their own dev notes/spec text, despite the project having already captured the correct answer in memory (`generate-output-dir-is-specscribeoutput`). Review caught it once but it kept resurfacing in later story files.

## Key Insights

1. Structural scope growth (new page types, new information architecture, new interaction models) needs a story-boundary decision, not another numbered polish pass — visual/CSS iteration is fine to absorb same-story, but a Kanban board redesign is not "polish."
2. When two stories' commits interleave in the working tree, disclose the cross-contamination in the *carrying* story's own File List/Completion Notes at review time — 2.5 did this correctly, after the fact; catching it earlier would avoid the ambiguity entirely.
3. A memory note existing doesn't stop a stale fact from being re-authored — story creation should actively check recent/adjacent story files for repeated stale commands or facts already captured in memory, not just rely on the memory existing.
4. Fixing the *process* gap from a prior retro (Action Item #2: retro cadence) doesn't guarantee the *content* gap gets fixed too (Action Item #1: the debt item itself) — both need explicit tracking, not just one.
