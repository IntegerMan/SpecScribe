## What Went Well

- **Story 10.10 (Context-Aware Navigation Bar)** was the epic's standout for the Project Lead — non-visual, high-value work: the existing white nav bar became page-type contextual (epic pages surface their stories, code pages surface siblings, etc.) instead of only working on Home. Quiet but exactly the kind of polish that makes the portal feel considered rather than assembled.
- **Seam reuse compounded across the epic**: 10.1's `RenderNavMarkup` seam fed both 10.10 and 10.11; 10.2's `ChartMeta`/`Framed` chart-frame fed 10.6 directly; 10.8's new `ListRow` primitive fed 10.9's sort/filter work. Each story left a hook the next one used for free — the "extract, don't duplicate" discipline held up under real pressure.
- **Epic 9's retro commitment was honored**: Story 10.7 explicitly locked its sunburst membership + click-destination contract at create-story time (per Epic 9 Action Item), and never invented a parallel navigation scheme.

## Challenges

- **Golden-fingerprint instability** was a standing tax from Story 10.2 onward — nearly every story had to reconfirm a regenerated content hash was stable across repeated runs before locking it in.
- **Shared-`main` concurrent-edit interference** hit roughly 5 of 11 stories (background auto-committer activity, concurrent sessions), and one incident escalated to an actual `git reset --hard` that transiently wiped uncommitted work mid–Story 10.8. This was a known, already-documented hazard going into the epic and still cost real work.
- **Story 10.10 required three post-"done" revision rounds** driven by live owner feedback (overflow disclosure, label truncation, a dark-bar auto-open bug) — a reminder that "non-visual" stories can still carry significant iteration cost.
- Several narrow technical-debt items remain open: `KnownIndexGroups` missing `adrs`/`retros` (open since Epic 4), a record-equality dedup bug shared by 10.8/10.10, 10.7's dense-collapse legend gap, and a handful of parser edge cases (10.3–10.5).

## Key Discussion Points (beyond the standard retro scope)

1. **Concurrent shared-main edits are a fact of this project's working environment, not a bug to engineer away.** The Project Lead's primary machine cannot run parallel git worktrees, so isolation isn't the fix. The team should default to assuming other agents may be editing the same shared code concurrently, treat that as acceptable, and verify state after edits rather than trusting isolation that doesn't exist.
2. **Possible future architecture reconsideration (parking lot, not an action item):** the Project Lead is unsure static HTML pre-rendering is still the right default and is considering whether a CLI that emits JSON consumed by a Vue-style (or similar) frontend should become the primary delivery model. Note: this wouldn't start from zero — ADR 0006 already ratified an *additive* JSON+SPA delivery adapter (Story 6.7), sitting alongside the static-HTML baseline with a noscript fallback. The open question is whether that adapter should become primary rather than additive. Flagged for a future correct-course session; not decided or scheduled here.
3. **ADRs are under-created.** The team confirmed this is a missing-trigger problem, not a quality problem — ADRs that do get written (0005, 0006) are solid, but structural decisions are consistently resolved as owner-locked notes embedded in story files / sprint-status.yaml prose instead of becoming a real ADR, unless the Project Lead explicitly asks for one. Recent examples that arguably should have been ADRs: 10.1's nav taxonomy, 10.6/10.7's owner-locked chart designs, 10.8's ListRow scope boundary.
