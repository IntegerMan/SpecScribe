# Story 3.6: Refinement Funnel on the Dashboard

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a stakeholder assessing planning maturity,
I want a funnel graphic showing epic → story → task refinement on the home page,
so that I can see how far the project has been refined at a glance.

## Acceptance Criteria

1. **Given** epics, stories, and task plans have been parsed **When** I view the dashboard **Then** a funnel visualizes the refinement stages (epics → drafted stories → stories with a task plan → tasks) with counts at each stage **And** the narrowing communicates how much detail work remains. [Source: epics.md#Story 3.6, lines 499-517]
2. **Given** the accessibility and truthfulness conventions established in Stories 1.4 and 1.5 **When** the funnel renders **Then** each stage carries a text label and value (never color-only) and reduced-motion is respected **And** an empty or early-stage project renders a sensible funnel rather than a broken or misleading one. [Source: epics.md#Story 3.6, lines 513-517; epics.md#Story 1.4/1.5]

  - [x] Subtask 1.3: **Truthfulness (AC #1 "narrowing", AC #2 "not misleading", Story 1.5 "no chart overstates progress") — the one non-negotiable constraint of this story.** These counts do **not** monotonically shrink — a healthy project has more stories than epics and far more tasks than stories, so a naïve "funnel that narrows by count" is either impossible (the data widens) or a lie. **You may choose either silhouette** (product owner deferred the visual reading to the developer), as long as it never overstates progress and every stage carries a real count as text:
    - **Option A — honest elaboration pyramid (simplest, recommended default):** width ∝ count, normalized to the largest stage (`maxStage = Math.Max(Math.Max(p.EpicsTotal, p.StoriesTotal), Math.Max(p.StoriesWithArtifact, p.TasksTotal))`). Reads few-epics → many-tasks. Communicate "detail work remaining" via per-stage **coverage sub-labels** (stage 2: "N of M epics broken into stories" = `p.EpicsDrafted`/`p.EpicsTotal`; stage 3: "P of S stories have a task plan" = `p.StoriesWithArtifact`/`p.StoriesTotal`), not via bar width.
    - **Option B — coverage funnel that narrows:** each stage's width = the fraction of the *prior* stage carried forward (% of epics with stories, then × % of stories with a plan), giving the classic narrowing look. If you pick this, you must define an explicit, defensible rule for the final "tasks" stage (it has no natural denominator) and document it in the builder's doc-comment — do not invent a ratio that makes tasks look smaller than reality.
    - **Forbidden either way:** scaling any stage's bar *smaller* than its true proportion just to force a narrowing silhouette — that is the exact misleading-chart failure AC #2 prohibits. The absolute count text is always shown; width is a supporting cue.
  - [x] Subtask 1.4: Guard the degenerate / early-stage cases so nothing renders broken or NaN (AC #2): if `p.EpicsTotal == 0` return the shared `"<div class=\"chart-empty\">…</div>"` placeholder (match the copy style of `Sunburst`'s `"Nothing to chart yet."` at `Charts.cs:130`). When `maxStage == 0` never divide by it (guard the width math). A stage whose count is `0` still renders its labeled row (count `0`), and a nonzero stage must get a small **minimum width floor** so a count of 1 next to a max of hundreds is still a visible, hoverable bar rather than a hairline.

## Dev Notes

- **This is a pure presentation story — no parsing, model, or git work.** Every number the funnel needs is already computed once per run by `ProgressCalculator.Compute` and carried on `ProgressModel` ([ProgressModel.cs](../../src/SpecScribe/ProgressModel.cs), [ProgressCalculator.cs](../../src/SpecScribe/ProgressCalculator.cs)) and threaded into `RenderIndex` → `AppendDashboard` ([SiteGenerator.cs:530](../../src/SpecScribe/SiteGenerator.cs), [HtmlTemplater.cs:210](../../src/SpecScribe/HtmlTemplater.cs)). Do not touch `ProgressCalculator`, `EpicsParser`, `TaskListParser`, or `GitMetrics`. Touch surface is exactly: `Charts.cs` (builder), `HtmlTemplater.cs` (one call site), `assets/specscribe.css` (styles), and the three test files.
- **Reuse the field the dashboard already calls "stories with a task plan."** `p.StoriesWithArtifact` is the count of stories that have a resolved implementation artifact ([ProgressCalculator.cs:21-24](../../src/SpecScribe/ProgressCalculator.cs)). The dashboard already presents this as "with a task plan" ([HtmlTemplater.cs:219-223](../../src/SpecScribe/HtmlTemplater.cs)) and the epic overview as "Stories detailed"/"with a task plan" (`EpicsTemplater.cs:131,469`). The funnel's stage-3 count is `p.StoriesWithArtifact`, full stop — matching the rest of the dashboard is more important than a purer definition, and a divergent count here would read as a bug.
- **The core design tension — read this before drawing anything.** The four counts *grow* down the pipeline in any real project (this repo today: ~7 epics, dozens of stories, hundreds of tasks). A classic "funnel narrows top-to-bottom by count" is therefore either impossible (the data widens) or a lie. The product owner has **deferred the silhouette choice to you** (elaboration pyramid vs. coverage funnel — both spelled out in Subtask 1.3), with one hard rule: whatever you draw must not overstate progress, and every stage must show its real count as text. This serves AC #1's "narrowing communicates how much detail work remains" *without* violating Story 1.5's non-negotiable "no chart overstates progress." If you find yourself scaling bars so later stages look *smaller* than their true proportion just to force a narrowing silhouette, stop — that's the exact misleading-chart failure AC #2 forbids. Option A (honest pyramid + coverage sub-labels) is the recommended default because it needs no invented denominator.
- **Charts are pure SVG/HTML + CSS variables, no JS** — deliberate, established convention, not an oversight ([[charting-is-pure-svg-no-js]], `Charts.cs:6-8`). A small tooltip/copy `specscribe.js` exists but the funnel needs no script. Do not add JS for the funnel.
- **Status color is single-sourced.** The six `--status-*` tokens ([specscribe.css:34-40](../../src/SpecScribe/assets/specscribe.css)) are the only place stage→color is defined; route the funnel's colors through `var(--status-*)` via `.funnel-*` classes and never hardcode hex ([[specscribe-status-token-system]]). This keeps the funnel consistent with the sunburst (`.sb-*`, css 1109-1114), donut (`.donut-seg.*`, 1466-1471), and progress bars (`.progress-fill.*`, 1450-1452) already on the same page.
- **Reduced-motion is already architected** ([[story-1-4-a11y-seams-for-1-5]]): entrance animations live in the `no-preference` block and are globally neutralized by the `reduce` block ([specscribe.css:2120-2147](../../src/SpecScribe/assets/specscribe.css)). Add the funnel to that seam (Subtask 3.2) instead of writing a new animation — the "reduce" half then covers you for free. AC #2's reduced-motion requirement is satisfied by following the existing pattern, not by inventing one.
- **Invariant formatting matters.** Use `Charts.F(double)` for any SVG numeric attribute (it formats `"0.##"` with `CultureInfo.InvariantCulture`, `Charts.cs:628`); a raw `double.ToString()` under a comma-decimal locale (de-DE, fr-FR) emits `"12,5"` and breaks the SVG. This is the same discipline the whole `Charts` class already follows.
- **Empty/early-state is a first-class requirement, not an edge case** (AC #2). SpecScribe explicitly targets "a project that's just getting started" (`Charts.cs:8` class doc) — a repo with 2 epics, 1 drafted story, 0 task plans, 0 tasks must render a coherent labeled funnel, not a blank box or a single sliver. Test this (Subtask 4.2).
- **Architecture context:** there is no `architecture.md`; the closest artifacts are [ARCHITECTURE-SPINE.md](../../_bmad-output/specs/spec-specscribe/ARCHITECTURE-SPINE.md) and [rendering-architecture.md](../../_bmad-output/specs/spec-specscribe/rendering-architecture.md), which describe a *target* shared-core/adapter seed (e.g. `IInsightProvider`, `SpecScribe.Core`) that **does not exist yet** — the current codebase is a single monolithic `src/SpecScribe` project. Per the spine's "Seed, Not Invariant": the monolith can be refactored later as long as the shared-core contract stays intact. Do **not** introduce the seed package/interface layout for this story — extend `Charts.cs` in place, matching the code's actual shape. [Source: 3-1 story Dev Notes; ARCHITECTURE-SPINE.md#Seed, Not Invariant]
- **No dedicated UX/PRD spec for the funnel.** Story 3.6 is a UX-forward addition seated under Epic 3 (Insight Surfaces); the only authoritative source is the epics.md AC (lines 499-517). There is no `EXPERIENCE.md`/`DESIGN.md` funnel entry and no PRD FR that pins the visual — hence the design guidance above is derived from the AC plus the binding Story 1.4/1.5 accessibility+truthfulness conventions it references. (Confirmed by full-text search of `_bmad-output` for "funnel"/"refinement".)
