## Story

As a project reviewer,
I want interactive tree views of directory and artifact structure,
so that I can inspect project organization and navigate to relevant content fast.

## Acceptance Criteria

1. **Given** a generated portal with multiple artifact families **When** I open the tree-view surface **Then** I can expand and collapse nodes by depth **And** each node has clear visual hierarchy cues and labels. [Source: epics.md#Story 3.4; PRD FR14; UX-DR19]
2. **Given** I use keyboard and screen reader navigation **When** I traverse the tree **Then** tree items are focusable with announced state (expanded or collapsed) **And** selecting a node can route to the related page or context target. [Source: epics.md#Story 3.4; PRD FR14; UX-DR19; NFR6]

## Dev Notes

- **This is the interactive-tree half of CAP Epic 3, explicitly handed off from Story 3.3.** Story 3.3's scope-boundary note is unambiguous: "Interactive/expandable tree views of the directory or artifact structure are **Story 3.4**, not this one. This story is a flat coverage/freshness panel, not a tree." [Source: [3-3-agent-and-workflow-structure-coverage-insights.md:55](../../_bmad-output/implementation-artifacts/3-3-agent-and-workflow-structure-coverage-insights.md)]. Story 3.4 delivers the actual tree surface. Build it as a **new first-class page** (`structure.html`) reached from nav + a dashboard quick link — the same page-plus-nav-gate shape [WriteSprint](../../src/SpecScribe/SiteGenerator.cs:581)/[SiteNav Sprint](../../src/SpecScribe/SiteNav.cs:97) already established for `sprint.html`. FR14 wording confirms a dedicated surface: "Provide project **tree views** and structural visualizations in generated outputs so users can inspect directory/layout shape and trace where planning and implementation artifacts live" [Source: [epics.md:45 (FR14)](../../_bmad-output/planning-artifacts/epics.md)].
- **The zero-JS decision is the single most important architectural call in this story.** Native `<details>`/`<summary>` provides ALL of AC #1's expand/collapse and AC #2's "focusable with announced state (expanded or collapsed)" with no JavaScript. The browser announces the expanded/collapsed state of a `<summary>` natively to screen readers, and Enter/Space toggle it. This is why you must NOT reach for a WAI-ARIA `role="tree"` roving-tabindex widget — that pattern *requires* JS keydown handlers this project deliberately forbids for chart/insight surfaces. The precedent is already in the codebase three times: [.story](../../src/SpecScribe/assets/specscribe.css:881) cards, `.requirements-inventory`, and [.dev-agent-details](../../src/SpecScribe/assets/specscribe.css:1163) all use native `<details>` with a custom `▸`/`▾` caret and zero JS. Follow that pattern exactly. [Source: [charting-is-pure-svg-no-js], [story-1-4-a11y-seams-for-1-5]]
- **Keep `Build` pure; keep IO in the generator.** `ProgressCalculator.Compute` and `WorkInventory.Build` (and, when Story 3.3 lands, `ArtifactCoverage.Build`) are all pure functions over already-parsed inputs; the `SiteGenerator` owns the disk reads. Follow the same split: `SiteGenerator` already enumerated the source `*.md` files ([EnumerateSourceFiles](../../src/SpecScribe/SiteGenerator.cs:697), `sourceRelatives` at [line 42](../../src/SpecScribe/SiteGenerator.cs)) and already holds `_docs` (source→output→title), so it builds the `outputHrefBySourcePath` map and passes it in. `ProjectTree.Build` does **zero disk access** — it is a pure function of (paths, href map) so every nesting/ordering/collapse rule is unit-testable without a repo on disk.
- **Data source = the existing source-artifact file set, not a fresh directory walk.** Do NOT add a new `Directory.EnumerateFiles` pass or read the output directory. The tree represents the **artifact structure** the portal already knows about — the `_bmad-output/**/*.md` set the generator enumerates once and renders into pages. Reusing that set (a) keeps the tree consistent with what's actually rendered, (b) avoids a second filesystem scan under the responsiveness budget (NFR1), and (c) means every navigable leaf already has a known generated page in `_docs`, so links can never dangle (Story 1.1 "no broken navigation"). ADRs live outside `_bmad-output` — they are a known scope exclusion here (they already have their own `adrs/` nav + landing); do not special-case them into this tree in v1.
- **Routing is best-effort, never a broken link.** AC #2 says selecting a node "*can* route to the related page or context target" — deliberately permissive. A file that maps to a generated page becomes an `<a>`; a file with no generated page (or a pure directory container) renders as a plain, non-link label. This mirrors the whole codebase's "linkify only when the target resolves, otherwise plain text" discipline ([SiteNav](../../src/SpecScribe/SiteNav.cs:5-7) "a missing doc is simply omitted rather than producing a broken link"; [RequirementLinkifier]/[SourceLinkifier] unresolved-id handling). Never emit an `<a>` to a page you didn't write.
- **Structure/hierarchy is NOT a lifecycle status — keep it off the `--status-*` tokens.** The six `--status-*` tokens are the single source of truth for lifecycle stage color and "planned work must never read as finished" [Source: [StatusStyles.cs:3-5](../../src/SpecScribe/StatusStyles.cs), [specscribe-status-token-system]]. A directory or a file node has no lifecycle stage. Style depth/branch/leaf with neutral palette tones (`--ink`, `--ink-light`, guide-rule borders), mirroring the same "keywords are grammar, not lifecycle stages" CSS discipline at [specscribe.css:953](../../src/SpecScribe/assets/specscribe.css:953). Do not invent new `--status-*`-shaped tokens.
- **Reuse `Icons` as the one glyph seam.** The tree needs a folder/structure glyph. Add it once to [Icons.ForConcept](../../src/SpecScribe/Icons.cs:31) under the `"Structure"` key (the same label the nav item + quick link use) — never a second glyph vocabulary. [RenderNavBar](../../src/SpecScribe/SiteNav.cs:145) auto-prefixes every nav item with `Icons.ForConcept(label)`, so adding the nav label `"Structure"` *requires* the glyph key to exist or the nav renders label-only (graceful, but you want the glyph). Every icon stays decorative + text-paired ([Icons.cs:3-11](../../src/SpecScribe/Icons.cs) contract; NFR6).
- **No motion — Story 3.5 owns insight animation.** Keep the tree static so reduced-motion is trivially satisfied (matching every existing chart and disclosure). Do not add expand/collapse transitions; that "flashy but purposeful visual language" is deliberately Story 3.5's scope. This is the same scope-discipline Story 3.3 practiced (it kept its panel static and left motion to 3.5). [Source: [epics.md#Story 3.5](../../_bmad-output/planning-artifacts/epics.md:479)]
- **Nav gate and page write are one signal.** The single failure mode to avoid is a `Structure` nav link to a `structure.html` that wasn't written. Gate BOTH on the same `StructureAvailable` boolean, exactly as `SprintAvailable` gates both `SiteNav.Build(hasSprint:)` and `WriteSprint` ([SiteGenerator.cs:576, 581, 719](../../src/SpecScribe/SiteGenerator.cs)). Watch mode calls `WriteIndex` on single-file edits but not `WriteStructure`; that's acceptable for v1 (the tree is stable across leaf-content edits — it only changes on add/rename/delete, which take the full-rebuild path [AD-5](../../_bmad-output/specs/spec-specscribe/ARCHITECTURE-SPINE.md:66)). Do not add incremental tree regeneration in this story.
- **No `architecture.md` exists** for this project — the governing docs are the spec kernel ([SPEC.md](../../_bmad-output/specs/spec-specscribe/SPEC.md), [ARCHITECTURE-SPINE.md](../../_bmad-output/specs/spec-specscribe/ARCHITECTURE-SPINE.md), companions). Per the spine's "Seed, Not Invariant" section the monolithic `src/SpecScribe` project is intentional — do NOT introduce the aspirational `IInsightProvider`/`SpecScribe.Core` split [Source: [ARCHITECTURE-SPINE.md:98-100](../../_bmad-output/specs/spec-specscribe/ARCHITECTURE-SPINE.md)]. Add `ProjectTree.cs` in place, matching the shape of `WorkInventory`/`ArtifactCoverage`/`ProgressCalculator`.
- **Windows/encoding + invariance gotchas already solved — reuse them.** Normalize every path through [PathUtil.NormalizeSlashes](../../src/SpecScribe/PathUtil.cs:10) and compare `OrdinalIgnoreCase` (Windows/Git-Bash path-case parity). Ordering must be `OrdinalIgnoreCase` (culture-independent — the same reason `GitMetrics`/`Charts` use `InvariantCulture`), so the output is byte-stable for the fidelity/snapshot tests. HTML-escape all labels/hrefs via [PathUtil.Html](../../src/SpecScribe/PathUtil.cs:24).
