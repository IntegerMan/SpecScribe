## What Went Well

- **The read-only invariant never cracked** across twelve stories, a new extension host, helper actions, editor bridges, tree view, status bar, and diagnostics — every review re-verified "no source planning artifacts mutated" (ADR 0005 AD-1/AD-2, ADR 0003 AD-6).
- **Spike-led de-risking earned the mid-epic flexibility.** 6.3 proved the webview seam before the 6.4 runtime; 6.6 measured delivery architecture (JSON+SPA+npx vs. bundled binary) with real numbers before committing — producing ADR 0005 and ADR 0006 from evidence, not opinion. 6.6 specifically stopped a pure-TS rewrite on vibes and made it a decision.
- **Golden byte-identity held the whole way.** The webview and SPA adapters were additive; the HTML surface never drifted — the Epic 4 guardrail paying off again through a greenfield surface addition.
- **Host-aware theming without semantic collapse (6.5).** The acceptance auditor confirmed webview chrome maps to VS Code host variables while SpecScribe's status/accent semantics stay distinct from host severity colors — the hard part of native theming, done right.
- **Owner-directed mid-epic scope was a strength, not creep.** Stories 6.8–6.12 were a deliberate correct-course to seat the native-integration recommendations, cheap precisely because the extension seam (6.3) and delivery decision (6.6) were already proven.
- **Continuity win — Epic 4 Action Item #2 landed.** Story 6.1's view-model contract read Story 4.1's ingestion-vs-delivery (AD-1 vs AD-2) distinction so naming didn't collide with `IArtifactAdapter`; 6.1 shipped clean through review.

## Challenges

- **Documentation drift between `epics.md` and `sprint-status.yaml`.** The Epic 6 renumbering + spike insertions + Stories 6.8–6.12 landed correctly in `sprint-status.yaml` (now explicitly "the operational truth"), but `epics.md` still numbers host-theming as its 6.3 and has no spike entries. For a tool whose job is to surface planning artifacts truthfully, a stale `epics.md` on its own epic structure is a dogfooded version of the drift SpecScribe exists to prevent. **Root cause:** agents update `sprint-status.yaml` on renumber/scope-change but never write the change back to `epics.md` — only one artifact was on the update path.
- **The "one F5 smoke in real VS Code remains" deferral recurred on 6.4, 6.8, 6.9, and 6.10.** Each deferral was individually correct (the CLI environment genuinely cannot paint Electron pixels), but four deferrals of the *same* gate means the TS shim → spawn → `webview.html` → paint path has only ever been reasoned about, never watched live. ADR 0005 flags this honestly under "Not yet proven." The C# side is proven exhaustively (headless render, CSP policy, packaging sizes/latency, 785 tests, golden byte-identity).
- **Byte-identity discipline silently carried a known defect forward.** Story 6.2 relocated `KnownIndexGroups` from `HtmlTemplater` into `DashboardViewBuilder` **byte-for-byte**, as the golden gate demanded — which meant Epic 4's still-open Action Item #1 (`adrs`/`retros` omitted from the well-known folder set, so a normal BMad repo permanently trips an "unrecognized top-level folder" notice) rode along untouched. Fixing it there would have broken the golden gate; the guardrail and the carried defect are in genuine tension.
- **Pre-existing `GoldenContentFingerprint` drift** (from in-progress Story 6.12 / spec-comment-block work) appeared as review noise across the 6.9–6.11 close-outs, repeatedly needing to be traced and dismissed as "not this diff."

## Key Insights

1. **Mid-epic scope addition is safe in proportion to how well the underlying seam is proven.** Epic 6's 6.8–6.12 wave worked because 6.3 and 6.6 de-risked the surface first. The lesson isn't "avoid mid-epic scope" (the owner's deliberate working style on personal projects) — it's "front-load the spike so the flexibility is cheap."
2. **A guardrail that enforces byte-identity can also preserve a known bug.** When relocating code under a byte-identity constraint, a carried-forward defect needs an *explicit* separate story with the golden-diff drill — it cannot be fixed inline without tripping the gate.
3. **Surface coverage should be an epic-level design input, not a per-story review afterthought.** Epic 6 grew from one HTML surface to three render adapters (HTML/webview/SPA) plus the extension host. "Does this feature reach every surface?" was proven per-story by `RenderParity`, but never scoped up front — which is exactly what the new integration-spike rule (Action #3) fixes.
