# Story 21.3: Planning ↔ Code Impact Map

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As someone connecting plans to reality,
I want to see which code areas each epic/story actually touched,
so that "what did this work change" becomes visible instead of inferred.

## Acceptance Criteria

1.
**Given** Epic 7 code citations and git commit→file data attributed to stories/epics
**When** the impact map renders
**Then** it correlates planning items with the code areas their commits touched (e.g., epic → touched files / areas), navigable to both the story and the code pages, reusing Epic 19's edges rather than a second schema.

2.
**Given** no commit-to-story attribution available (NFR8)
**When** the correlation can't be built
**Then** the surface is omitted cleanly
**And** it never re-counts open items against ProjectCounts.

## Context & Scope

Epic 21 gives first-time visitors and stakeholders a few high-impact displays that make the product's value legible at a glance. Story 21.1 built the traceability matrix (requirement × epic coverage); Story 21.2 (also not yet implemented as of this writing — `ready-for-dev`) is the delivery-cadence/cycle-time view. **Story 21.3 is the third: a planning↔code impact map** — for each epic (and, per story, each story), the set of code files its work actually touched, correlated at generation time and linked both ways (planning page ↔ code page).

**This story's AC text names two aspiration sources it cannot literally satisfy as written, and both were resolved with the owner before any code was planned (see "Owner-elicited decisions" below):**

1. **"Reusing Epic 19's edges rather than a second schema"** — Epic 19 (`19.1` Work-Graph Model spike, `19.2` Directed Graph Visualization) is **`ready-for-dev`, zero code landed** (verified against `sprint-status.yaml` and the epics.md text at time of writing). There is no work-graph edge model to reuse. This story does **not** wait on Epic 19 — it builds its own narrow, single-purpose epic/story → file correlation directly (mirroring how 21.1/21.2 did not block on their nominal siblings either). A coordination note is left for whichever Epic 19 story lands later to fold this correlation into its "cites"/"touches" edge kind rather than inventing a third parallel mapping — this story is NOT that generalized graph and must not try to become it.
2. **"Epic 7 code citations ... attributed to stories/epics"** — Epic 7's `[Source: ...]` citation mechanism (`CodeReferenceScanner`) is real and already ships (Stories 7.1/7.2), but it is a **declarative annotation** authored *inside* SpecScribe's own story/epic markdown files — a convention this project itself follows, but not one any arbitrary team's codebase already has. It was explicitly **rejected as the primary source** for this story (see decision below) because the goal is "help teams with the code they already have, with few modifications" — i.e., correlate using signals that already exist in ordinary git history (commit messages, branch names), not a new authoring requirement.

### Owner-elicited decisions (locked at create-story — do not re-litigate)

Elicited up front per the standing project rule that a new visual surface (and, here, a real data-source fork) gets its silhouette chosen by the owner, not the dev (memory `create-story-elicit-visual-intent`; precedent Stories 9.2, 7.12, 21.1, 21.2):

1. **Data source — git commit + branch-name mining, not `[Source:]` citations, not a required convention.** *Owner's own words: "We need to do something oriented at helping teams with the code they already have with few modifications. This means relying on commit parsing heavily or potentially branch name parsing for branches integrating into the main branch. We very much should avoid things like custom annotations and requiring different code or commit behavior while using the tool."* This means:
   - The correlation is **heuristic and best-effort**, mined from commit subjects/bodies and merge-commit branch names that already exist in this (or any) repo's history — **never** a new required annotation, commit-message convention, or authoring schema a team must adopt to use this feature.
   - `[Source: ...]` citations are explicitly **out of scope** as a data source for this story (they remain Epic 7's mechanism for a different purpose — linking prose to code — and are not reused here). Do not build a second citation-scanning pass for this feature.
   - Coverage will be **partial by construction** — many commits (in this repo and any other) won't mention a recognizable epic/story number, and that is the expected, honest outcome (AC #2), not a bug to work around with a fallback heuristic.
2. **Placement — dedicated page, plus a widget at the epic level and the story level (no dashboard strip).** Unlike 21.1/21.2's "dedicated page + dashboard strip" pattern, the owner asked specifically for: **(a)** a dedicated `impact-map.html` page (the full correlation, epic-grouped) and **(b)** a compact widget embedded directly on each epic's own detail page **and** each story's own detail page (not the home dashboard). This mirrors the `NextStepsHtml`-style pre-rendered-fragment precedent already used on `EpicPageView`/`StoryPageView` (Story 6.2 section-view-model discipline) rather than the dashboard-strip precedent 21.1/21.2 used.

### The data-mining design (validated against this repo's own real history — read before writing regex)

This repo's own commits and branches were inspected while writing this story, and the findings below are the concrete basis for the heuristic. **Do not re-derive this from scratch — read this section first.**

- **The shared fetch to reuse: `GitMetrics.TryComputeDeep(repoRoot)?.Commits`** (`IReadOnlyList<DeepCommit>`, [`GitMetrics.cs:459-482,123-129`]) — the SAME bounded (`-n 300`), one-shot `git log --numstat` fetch already powering Stories 3.2/3.8/7.4/7.5. Each `DeepCommit` already carries `Hash`, `Subject`, `Body`, and `Files` (repo-relative paths from numstat). **Do not add a second large git-log call** — this is the one shared fetch, per the project's established "one fetch, one parse, several views" rule.
- **Merge commits appear in this same fetch, with a real branch name in `Subject`, but ZERO `Files`.** Verified empirically against this repo:
  ```
  git log --numstat --pretty=format:<same shape as GitMetrics.cs:473> --grep="Merge branch"
  ```
  returns merge commits like `Merge branch 'worktree-bmad-dev-story-6-4'`, `Merge branch 'spike/delivery-arch-6-6'`, `Merge branch 'worktree-bmad-dev-story-3-8'` — **with no numstat rows at all** (git does not diff merge commits by default, and the existing fetch does not pass `-m`/`-c`). This means a merge commit's `Subject` is a genuine, zero-cost signal (an epic/story number spelled out in the branch name) but its own `Files` list is always empty — it can never itself supply the touched-file set.
- **Real branch names in this repo carry recognizable `epic-story` number pairs** (confirmed via `git branch -a` / `git log --merges`): `worktree-bmad-dev-story-6-4` → 6.4, `spike/delivery-arch-6-6` → 6.6 (confirmed against memory `adr-0005-reopened-delivery-arch-spike` — this *is* Story 6.6's spike), `spike/vscode-6-3` → 6.3 (Story 6.3's VS Code spike), `worktree-bmad-dev-story-3-8` → 3.8, `worktree-story-7-8-related-files-graph` → 7.8, `claude/bmad-create-story-7-1-caac2f` → 7.1. Others (`feat/code-review-single-story-id`, `docs/readme-frameworks-table`, `fix/ci-full-git-history-commit-activity`) carry **no** number at all — expected, must resolve to "no attribution," never a guess.
- **Many individual (non-merge) commit subjects in this repo already self-identify** without needing the branch at all: `"7.11"`, `"7.10 dev revisions"`, `"fix: Story 6.7 deferred-work cleanup ..."`, `"fix: Story 6.9 deferred-work cleanup ..."`. These are caught by direct per-commit subject/body parsing alone — no branch-name step required for them.
- **Many other real commit subjects in this repo carry NO identifiable number at all**: `"Code review on 7.x"` (a range, not a clean number — do not try to expand "7.x" into every 7.N story; treat as unresolved), `"Overnight code review"`, `"Other investigations"`, `"Adjustments"`, `"Plan work"`, `"Misc changes"`, `"Retro work"`. **These must resolve to "no attribution" — never guessed, never inherited from an unrelated nearby merge.** Spot-checked: `git log --oneline --first-parent -5 <a merge>` shows commits like `"Review notes"`/`"Code review"`/`"Decision-making and ADRs"` sitting directly on the mainline BETWEEN merges (not inside any merged branch) — there is no branch name available to backfill these, and none should be invented.
- **Design (two-tier, both bounded, no new large git call):**
  1. **Tier 1 — direct match.** For every `DeepCommit`, regex-scan `Subject` + `Body` for an epic/story mention (`Story\s+(\d+)\.(\d+)`, `Epic\s+(\d+)`, and a bare `(\d+)\.(\d+)` token) and **validate the parsed number(s) against the real roster** — `EpicsModel.Epics[].Number` and `EpicsModel.Epics[].Stories[].Id`. A parsed number that doesn't correspond to a real epic/story is discarded, never guessed into existence (this is what keeps `"Code review on 7.x"` from false-matching, and prevents an unrelated coincidental number from misattributing). A commit may validly match more than one identifier (this repo's own history shows commits that legitimately bundle multiple stories — memory `story-7-1-code-review-done`) — attribute to all validated matches, don't force a single winner.
  2. **Tier 2 — merge/branch backfill, linear-window approximation.** For a merge commit whose `Subject` (the branch-name text) resolves to a validated identifier via the same Tier-1 regex/validation, attribute that identifier to the run of **immediately-following, still-unattributed** commits in the (already fetched, already-ordered) `Commits` list, up to the *next* merge commit boundary (i.e., the commits that sit between this merge and the previous one in the log, which have not already resolved a Tier-1 match of their own). This is a **linear, single-pass, no-new-git-call approximation of "which commits this branch merged"** — it is correct for the common sequential/solo-branch pattern this repo's own history exhibits, and explicitly imprecise for repos with heavy concurrent branching (multiple branches interleaved before either merges). **State this limitation plainly wherever the feature's provenance/caveat text renders** — do not silently pretend it is exact ancestry. (A precise version would need each commit's parent hashes via `%P` and an in-memory DAG reachability walk — deliberately **out of scope**, see "Out of scope" below; this story's bar is "materially better than nothing, honestly labeled," not "exact.")
  3. Union the `Files` of every commit attributed (by either tier) to a given epic/story into that epic's/story's touched-file set (dedup by path). A story's touched files also roll up into its parent epic's set (an epic's touched files = the union over its own directly-matched commits' files ∪ every one of its stories' files).
  4. **Never fabricate, never guess, never silently drop the caveat.** A commit with no Tier-1 match and no Tier-2 backfill is simply excluded — this is the expected, common, honest outcome for most of this repo's real commits, not an error. The rendered surface must say, plainly, that this is a best-effort correlation derived from commit/branch naming, not a tracked or authoritative mapping (mirrors 21.2's cycle-time "approximate" caveat pattern).
- **Only link to files that actually have a code page.** Cross-check each attributed file path against the same code-page existence gate `CodeReferenceScanner`/`CodeReferenceLinkifier` already use (a file must exist, resolve inside the repo, and not be an ignored/process file) so the impact map never emits a dead link — reuse `CodeReferenceScanner.TryResolveRepoFile`/the existing code-page map, do not re-derive a third "is this a real linkable file" gate.
- **This is a real, additional data dependency beyond `hasEpics`.** `DeepGitPulse.Commits` is **only populated when `--deep-git` ran** (`GitMetrics.TryComputeDeep`, opt-in). A normal (non-deep-git) generate has `progress.DeepGit == null` — there is no commit/file data to correlate at all. Gate the page/nav/widgets on **`hasEpics && hasDeepAnalytics`** (`hasDeepAnalytics = progress?.DeepGit is not null`, the exact signal `SiteNav.Build`/`SiteGenerator` already compute for Deep Analytics — [`SiteGenerator.cs:183-184,4013-4014`]), not on `hasEpics` alone. This is a deliberate, justified departure from 21.1/21.2's "share the exact same `hasEpics` gate, never add a new flag" pattern — the two prior stories' data (`RequirementsModel`/`StoryInfo.LastUpdatedDate`) needs only `epics.md`; this story's correlation genuinely cannot exist without deep-git data, so gating on `hasEpics` alone would produce a nav entry/widget that 404s or renders empty for the (common) non-`--deep-git` case.

### Guardrails & invariants (must follow)

- **No new authoring convention.** Nothing about this feature requires a team to write commit messages or branch names any differently than they already do (owner's explicit instruction). A commit/branch that happens to already look like `"Story 6.7 ..."` or `"story-7-8-..."` is caught; nothing is required to make it look that way.
- **Validate against the real roster, never invent an ID.** Every Tier-1/Tier-2 candidate is checked against `EpicsModel`'s actual epic numbers and story ids before being trusted. A bare number that doesn't correspond to a real epic/story is discarded, not guessed into existence — this is what keeps a commit like `"Code review on 7.x"` or an arbitrary unrelated number from misattributing.
- **Tier 2 is a stated approximation, not real ancestry.** The linear "commits between this merge and the previous merge" window is deliberately not a `%P`-based DAG walk (see "Out of scope"). Its caveat text must render wherever the feature's data appears (page `Note` slot at minimum) — never presented as exact.
- **Bounded, one shared fetch.** Reuses the existing capped (`-n 300`) `DeepGitPulse.Commits` — no second large git-log call, no per-file/per-commit additional git shell-outs.
- **Never fabricate or silently drop the caveat.** Zero attribution is a normal, common, honestly-labeled outcome (most commits in most repos, including this one, won't self-identify) — never an error, never papered over with a guess.
- **Never a dead link.** An attributed file that doesn't resolve to a real, existing, non-ignored repo file (or has no code page) is dropped from the rendered list rather than linked.
- **Single-source status/counts untouched.** This story adds no new tally against `ProjectCounts`; it doesn't touch `StatusStyles` classification either (it's about code correlation, not lifecycle stage).
- **Section-view-model discipline (Story 6.2).** The epic/story widgets are pre-rendered fragment fields on `EpicPageView`/`StoryPageView`, built once, consumed identically across HTML/SPA/webview — never hand-appended in one adapter only.
- **Pure SVG/HTML + links, no info-bearing JS** (memory `charting-is-pure-svg-no-js`). This surface is plain linked lists, not a chart requiring JS at all.
- **Deterministic (NFR8/FR31/CI reproducibility).** File-set ordering is ordinal-sorted, not dictionary-iteration-order-dependent; no wall-clock use anywhere in this feature (the underlying commit data is already time-agnostic once fetched).
- **Degrade, don't break (NFR8).** No `--deep-git` → the nav entry and widgets are absent (not broken links, not empty panels). `--deep-git` on but nothing attributable → the dedicated page still renders with its honest empty note (mirrors 21.1's "coverage not yet mapped" precedent); epic/story widgets with nothing to show are absent, not empty (per Task 4).
- **Accessibility.** Everything inside `<main id="main-content">` on the new page; file lists are plain semantic `<ul>`/`<ol>` with real links (no info conveyed by an SVG alone); keyboard-reachable links, focus ring (Story 1.4).
