## What Went Well

- **The "one seam" discipline held for a third epic straight.** Every insight rode an existing single-source seam: `GitMetrics.RunGit`/`ParseLog` as the one git integration point (3.1/3.2/3.8), the six `--status-*` tokens, `Icons.ForConcept`, `ModuleContext.WellKnownDocs`, and `ArtifactCoverage` as the declared Epic-4 family seam. No story forked a second git module or a parallel palette.
- **Story 3.5 added a *parallel* token system rather than overloading an existing one.** The new `--motion-*` family mirrored the `--status-*` single-source discipline for timing, routed through the two paired reduced-motion media blocks, and delivered the interactive-legend emphasis in pure CSS (`:has()`) with **no new JS** — honoring the "charts are pure SVG/HTML, no JS" convention.
- **Progressive-enhancement JS landed cleanly where it was explicitly sanctioned.** Story 3.8's Git Insights hub is the one surface the rendering architecture names for client-side sort/filter; it shipped a no-JS-complete, server-sorted baseline first, with JS as pure enhancement (and the filter control *created* by JS so nothing dangles without it).
- **Graceful degradation stayed contractual.** The `--deep-git` gate *is* the FR-10 performance guarantee (deep path never runs when off); insight failures degrade to an absent panel/page, never a broken baseline (AD-4 upheld throughout).
- **Real latent bugs were caught and fixed, not just papered over.** Story 3.2 fixed renamed-file `=>` artifacts corrupting coupling; Story 3.7 fixed the `Epics 1 & 2` plural-coverage parse that was silently mis-marking every multi-epic FR as unmapped/Planned (FR2/FR5/FR7 now roll up correctly).

## Challenges

- **Owner-driven redesign-after-review was the epic's dominant rhythm — and it was rework, not iteration.** Story 3.1 gained a whole new AC #3 (panel consolidation) post-review; 3.2 grew a dedicated `deep-analytics.html` page + coupling graph in a "Task 6"; **3.3's panel was redesigned four times** (passive list → actionable grid → "Planning Artifacts" + meter → header meter); **3.6's funnel pivoted three times** (vertical pyramid → sideways funnel → cumulative Story Pipeline); 3.7 had a full round-2 acceptance reshape.
- **Root cause (owner-identified): create-story delegated visual direction to the dev instead of eliciting it from the owner.** Stories 3.6 and 3.7 wrote — verbatim — *"no dedicated UX/PRD spec pins this visual… the developer owns the silhouette."* Create-story **detected** the missing `DESIGN.md`/`EXPERIENCE.md` entry and, rather than asking the owner, handed the choice to the dev. The dev didn't guess badly; the process told them to guess. The Project Lead confirmed: "I'm having to correct the agent's paths on the UX on most cards; I'd prefer to be asked more questions during the create-story phase."
- **A recurring parser failure shape:** "the parser silently drops a real-world input shape." Three instances — renamed-file undercount (3.1 deferred → 3.2 fixed), the plural-`Epics` coverage drop (3.7). Epic 4 will feed *foreign* framework inputs into these same parsers.
- **Tracking lagged reality.** The Epic 1 heatmap debt was fully fixed via `spec-epic1-heatmap-debt-triage`, but its action item stayed `open` in `sprint-status.yaml` — Epic 2's Insight #4 ("fix the thing, forget to close the ticket") recurring.

## Key Insights

1. When a story has a visual surface with owner-latitude and no pinning spec, that is the *signal to ask the owner*, not to delegate the silhouette to the dev. The gap is detectable at create-story time — the story files literally name it.
2. "Split, don't absorb" (Epic 2 Action #2) was learned at the **epic-restructure** scale (3.4→7.6, 3.6→3.7 spin-out) but not yet at the **within-story** scale (the many post-review redesign passes).
3. Fixing a content gap and closing its tracking entry are two separate acts; a resolved item that stays `open` misreports project state indefinitely.
4. Every `FirstOrDefault`-picks-one assumption waved through as "this repo has exactly one" (spec kernel, `prd.md`, `sprint-status.yaml`, memlog) becomes a real multi-framework input in Epic 4 Story 4.2 — the assumptions come due all at once.
