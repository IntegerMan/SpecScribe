# Story 3.7: Requirements Flow and Status Blocks

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a stakeholder tracking requirements maturation,
I want each FR/NFR shown as a colorized status block and a Sankey-style flow of functional requirements from definition through epic coverage into implementation states,
so that I can see how requirements are maturing from definition to delivered at a glance.

## Acceptance Criteria

1. **Given** the requirements inventory has been parsed **When** I view the requirements page or the dashboard requirements panel **Then** FRs and NFRs render as a grid of colorized status blocks driven by the shared status tokens **And** each block carries its id and a text/tooltip status so state is never color-only. [Source: epics.md#Story 3.7, lines 525-529]
2. **Given** functional requirements trace into epics and stories **When** the requirements flow view renders **Then** a Sankey-style diagram shows FRs flowing from definition through epic coverage into implementation states, backed by a structured FRŌåÆstory mapping established by this story (extending the epic-level coverage map) **And** unmapped or deferred requirements appear as honest, labeled flows rather than being dropped. [Source: epics.md#Story 3.7, lines 531-535]
3. **Given** the accessibility, truthfulness, token, and motion conventions from Stories 1.4, 1.5, and 3.5 **When** these visualizations render **Then** they inherit those conventions in full (status tokens only, text alternatives, reduced-motion seams, no overstated progress). [Source: epics.md#Story 3.7, lines 537-540; epics.md#Story 1.4/1.5; 3-5-flashy-but-purposeful-insight-visual-language.md]

## Dev Notes

### The central data-honesty tension ŌĆö read before drawing anything

This story is the requirements-maturation visualization that **Story 3.6 originally drifted toward and was pulled back from**. When 3.6's funnel was redirected to "story pipeline / implementation progress," the requirements-maturation vision was spun out here with its scope corrected and its data gap named explicitly ([3-6 Change Log 2026-07-09](3-6-refinement-funnel-on-the-dashboard.md); [epics.md:511-517](../planning-artifacts/epics.md)). Two honesty constraints govern everything:

1. **FR coverage is epic-level, and status is a deliberately conservative roll-up.** `RequirementStatus` has only Deferred/Planned/Ready/Done and **intentionally omits "in development"** because one in-flight story cannot be pinned to a specific requirement through an epic-level map ([RequirementsModel.cs:5-10](../../src/SpecScribe/RequirementsModel.cs), [RequirementsParser.cs:137-158](../../src/SpecScribe/RequirementsParser.cs)). The Sankey's terminal column is exactly these four states ŌĆö do not fabricate a finer state. This is the same "no chart overstates progress" rule Story 1.5 pinned and 3.6 fought (its whole silhouette redesign was about it). If you find yourself claiming an FR is "in development," stop ŌĆö the data doesn't support it.
2. **The FRŌåÆstory mapping this story adds is real but bounded.** AC #2's "structured FRŌåÆstory mapping (extending the epic-level coverage map)" = capturing *all* covering epics per FR (Task 1.1, today's parser drops the 2nd epic of `Epics 1 & 2`) and resolving those epics to their stories at render time (Task 1.3). It makes the flow genuinely story-backed. It does **not** license per-story FR status claims (constraint 1). This mirrors how 3.6 was told "you may choose the silhouette, but never overstate progress" ŌĆö you have latitude on layout and the multi-epic routing rule, none on honesty.

### This is NOT a pure-presentation story (unlike 3.6)

3.6 touched only `Charts.cs` + one call site + CSS + tests because every number was already on `ProgressModel`. **3.7 requires a parser/model change** (Task 1) because the multi-epic FRŌåÆstory mapping does not exist yet. Budget for it. Touch surface: `RequirementsModel.cs` + `RequirementsParser.cs` (data), `Charts.cs` (two builders), `RequirementsTemplater.cs` (render + one signature change), `SiteGenerator.cs` (one call-site update), `assets/specscribe.css` (styles + motion), and the test files. **No new files or folders, no new project** ŌĆö extend the monolith in place (see architecture note below).

### Reuse, don't reinvent

- **Status ŌåÆ color is single-sourced.** `StatusStyles.ForRequirement` ([StatusStyles.cs:84-90](../../src/SpecScribe/StatusStyles.cs)) already maps a requirement to `done/ready/pending/deferred`; `RequirementLabel` gives the word. Route every block and flow node through these + `var(--status-*)`. Never hardcode hex. [[specscribe-status-token-system]]
- **HTML chart builders are a thing.** `EpicMosaic`, `ArtifactCoveragePanel`, `GitPulsePanel` return HTML from `Charts.cs`, not SVG ŌĆö the status-block grid belongs with them. The Sankey belongs with the SVG builders (`Sunburst`, `CouplingGraph`, `RefinementFunnel`).
- **`CouplingGraph` is your Sankey precedent** ([Charts.cs:975-1045](../../src/SpecScribe/Charts.cs)): generation-time layout, deterministic ordering, weighted strokes, `role="img"` name, per-element `<title>`, pure SVG ŌĆö no JS. **`RefinementFunnel`** ([Charts.cs:692-777](../../src/SpecScribe/Charts.cs)) is your stage-flow precedent (columns, connectors, per-stage labels/counts, token fills, empty guard). Between them you have every technique this story needs.
- **The requirements page already has the scaffolding** ŌĆö donuts, group navigator, req cards, per-requirement detail pages, linkification ([RequirementsTemplater.cs](../../src/SpecScribe/RequirementsTemplater.cs), [SiteGenerator.cs:850-868](../../src/SpecScribe/SiteGenerator.cs)). Add sections to `RenderIndex`; don't fork a new page. The existing req cards are the text equivalent that keeps the flow from being color/shape-only.

### Charts are pure SVG/HTML + CSS variables, no JS

Deliberate, established convention ([Charts.cs:6-8](../../src/SpecScribe/Charts.cs), [[charting-is-pure-svg-no-js]]). A small tooltip/copy `specscribe.js` exists but neither the grid nor the flow needs script ŌĆö compute the Sankey layout in C# at generation time (as `CouplingGraph` does) and emit static SVG. Do not add JS.

### Reduced motion is already architected

Entrance animations live in the `no-preference` block and are neutralized by the paired `reduce` block ([specscribe.css:2627-2712](../../src/SpecScribe/assets/specscribe.css)); `--motion-*` tokens are the single timing source ([specscribe.css:43-53](../../src/SpecScribe/assets/specscribe.css), [[motion-token-system]]). Join the seam (Subtask 4.3) ŌĆö key your new roots into BOTH halves ŌĆö rather than authoring a new animation. `.refinement-funnel` is the exact template to copy: it appears in the `no-preference` block with `panel-reveal` and in the `reduce` block with `animation: none`.

### Invariant formatting matters

Use `Charts.F(double)` for every SVG numeric attribute ([Charts.cs:1152](../../src/SpecScribe/Charts.cs)); a raw `double.ToString()` under a comma-decimal locale emits `"12,5"` and breaks the path/width. Dates (none expected here) would use `DReadable`. Same discipline the whole `Charts` class follows.

### Empty / early-state is a first-class requirement

SpecScribe explicitly targets "a project that's just getting started" ([Charts.cs:8](../../src/SpecScribe/Charts.cs)). A project with 2 FRs, one deferred, no completed epic must render a coherent grid and a coherent (if tiny) flow ŌĆö not a blank box, a hairline, or a NaN path. Test it (Subtask 5.3). This is AC #3's truthfulness applied to the low-data case.
