## What Went Well

- **The 4.3–4.7 → Epics 11–15 split was a planned readiness call, not rework.** Confirmed by the Project Lead: the team wasn't ready to do reliable dedicated work on the other frameworks yet. Each new epic's `X.1` Integration Spike explicitly requires surveying real repos of that framework before any coverage story is scoped — front-loading discovery the shared Epic 4 container couldn't have supported.
- **Epic 3 Action Item #1 (elicit visual intent at create-story) landed concretely.** Story 4.8 is the one story this epic that introduced a new visual surface, and its design direction (diagnostics-first table, config in a collapsible `<details>`, footer→About→Diagnostics reachability) was owner-locked up front, explicitly citing the Epic 3 retro action item by name.
- **Epic 3 Action Item #3 (split, don't absorb, at the within-story scale) landed concretely.** Story 4.8 was spun out of Story 4.2 mid-story specifically so diagnostics work wouldn't get absorbed as a subtask — the first clean instance of that principle applied within an epic rather than only at the epic-restructure scale.
- **Golden-diff discipline held exactly where it needed to.** 4.1 and 4.2 both proved byte-for-byte identical output on this repo's canonical layout; 4.8 knew explicitly, in its own scope block, that it was the sanctioned exception to that guardrail.
- **Review findings stayed sharp.** 4.1 caught an `OverflowException` on an oversized artifact number that would have violated the adapter's own "never throws" contract; 4.2 caught a status-pill inconsistency between an ADR's index card and its own page — exactly the edge-case class Epic 3's retro asked reviewers to hunt for.

## Challenges

- **A cross-story defect surfaced in review but was left unowned.** Story 4.8's review found that `HtmlTemplater.KnownIndexGroups` (Story 4.2's code) omits `adrs`/`retros` from the well-known folder set, so a normally-laid-out BMad repo permanently emits an "unrecognized top-level folder" notice for its ADR directory — undercutting 4.8's own all-clear-state promise on real projects. It was deferred as "pre-existing, raise against 4.2's review," but nothing picked it up. Resolved this retro: queued as its own future story rather than left to rot.
- **Two Epic 3 action items have no fresh evidence either way this epic.** Action #2 (close a tracking entry in the same change the fix lands) and Action #4 (a review-checklist item for "what real-world input shape does this parser silently drop?") stay open — Epic 4 didn't generate a clean test of either, since 4.1/4.2 only exercised BMad's own inputs through tolerant paths. The real test of Action #4 arrives with Epics 11–15's foreign-framework inputs.

## Key Insights

1. A planned scope extraction, made *before* committing to work the team isn't ready to do reliably, is a different (healthier) event than the redesign-after-review rework Epic 3 struggled with — worth distinguishing explicitly in future retros rather than lumping all mid-epic restructuring together.
2. Retro action items can land in a *later* epic than the one that generated them, in a story that wasn't obviously "about" the action item (4.8 is a diagnostics page, not a UX-process story) — action items should stay visible past the epic boundary that produced them.
3. A defect found in one story's review but attributed to a sibling story's code needs an explicit owner at discovery time, or it sits unassigned indefinitely — a deferred cross-story finding is a different risk class than a same-story deferral.
