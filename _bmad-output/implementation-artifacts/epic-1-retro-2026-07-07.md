**Date:** 2026-07-07
**Participants:** Matthew-Hope Eland (Project Lead), Amelia (Developer, facilitator), Alice (Product Owner), Charlie (Senior Dev), Dana (QA Engineer), Elena (Junior Dev)

## Epic Summary

- Delivery: 5/5 stories completed (100%) — 1.1 Dashboard Navigation & Readability Foundation, 1.2 Traceability Links, 1.3 Markdown Fidelity, 1.4 Accessible High-Polish Interaction Baseline, 1.5 Dashboard Insight Polish and Visual Truthfulness
- Test suite growth: 83 → 127 → 164 → 176 → 197 tests (net +114, zero regressions across the epic)
- Quality process: every story closed with a 3-layer adversarial code review (Blind Hunter, Edge Case Hunter, Acceptance Auditor); real production defects only began surfacing in Story 1.5, and were patched same-story
- Scope events: Story 1.4 was split mid-planning into 1.4 (accessibility/motion) + new Story 1.5 (polish/truthfulness), with further items routed to Stories 2.1, 3.1, 3.5; Story 1.3 grew a third AC (TOC sidebar) mid-story by product-owner direction
- This was the project's **first retrospective** — no prior retro to cross-reference

## What Went Well

- **Seam preservation held across the epic.** Story 1.1 established central seams (`SiteNav`, the linkifier/reference-map pipeline); every later story extended rather than forked them, producing a "verify-and-harden" pattern (1.2–1.5 each note large portions of ACs already satisfied at baseline).
- **Scope-splitting was deliberate, not reactive.** When Story 1.4 started absorbing the full UX review, the team split it by concern (accessibility vs. polish vs. work-representation vs. deep-analytics vs. flashy-visuals) into 1.4/1.5/2.1/3.1/3.5 rather than cramming it into one story.
- **Test discipline was continuous**, not an epic-end scramble — 114 new tests landed incrementally, one story at a time.
- **"Reinvention and Regression Guardrails" sections were genuinely load-bearing** — naming the specific thing not to rebuild (e.g., "do NOT build the JS drill engine") kept scope from creeping story to story.
- **Process win beyond any single story:** the team established a reliable Claude Code + BMad workflow, got an automated build pushing docs to GitHub Pages, generalized the tool away from being tied to one specific (GDS-shaped) project, and moved SpecScribe into its own repository — a prerequisite for Epic 4's multi-framework goals.

## Challenges

- **CI/publishing reliability was a hidden cost.** Getting a dependable GitHub Pages build was harder than expected — GitHub's documented workflow YAML didn't match what was actually needed, producing flaky builds that sometimes published and sometimes failed. This distracted from story work and required unplanned iteration/experimentation. No story file budgeted time for this; it surfaced reactively (e.g., Story 1.5's review found `fetch-depth: 0` was needed for real commit history, and added cache-busting query params after stale-CSS artifacts were traced to caching, not a source defect).
- **Retro cadence lagged actual delivery pace.** This retrospective is landing after Epic 2 is already 5/6 stories in `review` — the epic boundary in `sprint-status.yaml` didn't trigger a timely retro, so the "prepare for next epic" half of this process arrived too late to be useful for Epic 2.

## Resolved Since Discussion

- The CI/Pages publishing flakiness has since been fixed and documented (no longer an open item).
- CI/publish reliability has since been established as an explicit risk category in planning (no longer an open item).

## Key Insights

1. Central-seam discipline (don't fork, extend) is what made a "verify-and-harden" delivery pattern possible and kept regressions at zero across five stories.
2. Deliberate mid-epic scope-splitting, done early and by concern, is healthier than absorbing scope creep into one oversized story.
3. Infrastructure/CI work is invisible in story-level planning and can silently consume significant unplanned effort — it needs a place in epic-level risk awareness, not just in-story firefighting.
4. Retrospective cadence needs to track actual story completion, not epic boundaries alone, or the "next epic prep" function of a retro loses its value.
