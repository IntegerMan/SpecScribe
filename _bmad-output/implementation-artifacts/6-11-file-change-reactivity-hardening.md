## What this story is (and isn't)

This is **reactivity hardening**: three defects/gaps in *how the live view learns a source changed*, none of which touch rendering. It is the most **defect-adjacent** story in Epic 6's native-integration wave ‚Äî R6.1 is a **shipped live-data bug** ([R6.1](../../docs/VSCodeIntegrationRecommendations.md), line 85, and the epic AC #1 calls it "the most defect-adjacent item"). Three deliverables, mapped to the two epic ACs:

1. **AC #1 ‚Äî non-`.md` sources refresh the view (R6.1).** `sprint-status.yaml` (the sprint board / Now & Next data source) and `_bmad/config.toml` (project name) change and the view goes **stale silently today** ‚Äî blocked by `.md`-only watch globs (extension) and `Filter = "*.md"` (core). Fix in **BOTH layers**, and on the core side the fix is **more than the `Filter` property**: `.md` is enforced in **three places**, and the fire-time dispatch has **no route** for a yaml/toml change (its routes assume a markdown artifact). This is a **reviewed change, not a drive-by glob edit** (the epic AC says so verbatim).
2. **AC #2a ‚Äî watch roots derived from core-resolved roots (R6.2).** The extension watches literal `_bmad-output/` + `docs/adrs/` today; a repo with **non-default source/ADR roots** (Story 5.1/5.2) or **opened on a subdirectory** live-updates the **wrong paths**. Carry the **resolved** roots on the `webview` payload (the datum 6.10 punted here) and build the watchers from those ‚Äî **no hardcoded path literals in TS**. This is the **uniform fix** for the subdir-open caveat 6.10 flagged (watchers **and** reveal-source).
3. **AC #2b ‚Äî visibility-aware refresh (R6.3).** A hidden panel/tree should **mark itself dirty** while hidden and re-render **once on reveal**, not re-spawn a ~2 s render per change burst while nothing is visible.

### The five Epic 6 native-integration constraints ([¬ß2](../../docs/VSCodeIntegrationRecommendations.md))

1. **Thin shim, no rendering brain** (AD-1/AD-2). The extension parses **no** markdown/yaml/toml and derives **no** path structure. The **resolved roots are computed in C#** and carried in the payload (exactly like `configuredOutputRoot`); TS only builds a `RelativePattern` from them. If you write a `_bmad-output`/`docs/adrs`/`_bmad` literal into TS path logic, you've crossed AD-1/AD-2 ‚Äî push it into the core datum. The one allowed TS "logic" is joining a core-emitted relative path to the workspace folder.
2. **Read-only end to end** (AD-6, ADR 0003, FR-17, NFR-5). File watchers **take no locks and write nothing** ([FileWatcherService.cs:5‚Äì7](../../src/SpecScribe/FileWatcherService.cs): reads are always shared, never a write lock on the watched tree). This story changes **what** is watched and **how refresh is scheduled** ‚Äî it writes **no** project artifact, adds **no** settings-`.update`, and stages nothing. NFR5 (shared reads, no write locks on observed files) is a **hard invariant** this story must preserve for the newly-watched yaml/toml too.
3. **VS Code settings carry HOST concerns only** (ADR 0003). This story adds **no new setting** (R5.1's proposed `specscribe.refresh.debounceMs`/`.enabled` are **OUT** ‚Äî see Scope). Debounce stays the frozen 400 ms.
4. **The generated HTML surface stays byte-identical.** Nothing here touches the generated site: the extension change is TS-only; the core change is (a) a widened watcher + a new dispatch **route** (watch-mode behavior, not page markup) and (b) an **additive payload datum** (resolved roots) ‚Äî neither emits HTML. The golden `GoldenContentFingerprint` is unaffected by construction (memory: [golden-diff-normalization-gotchas]). Re-run it anyway (Task 6).
5. **Status stages, not host severities.** Not exercised by this story (no status iconography here); the constraint stands but there is nothing to route.

## Design decisions captured at create-story

### AC #1 core: "The new data-source route" ‚Äî re-parse and rebuild what yaml/toml feed

`sprint-status.yaml` and `config.toml` are **not** markdown artifacts; the four existing dispatch routes (`RegenerateAdrs`/`RegenerateEpics`/`GenerateOne`/`RemoveFor`) either ignore them or (worse, for sprint-status) mis-route to `RegenerateEpics` which won't re-parse sprint (fact #3). Add a **new route** the timer dispatch selects **before** the `IsEpicsRelated` check:

- **Classify data-source files first.** Add `SiteGenerator.IsDataSource(fullPath)` ‚Üí `true` for `sprint-status.yaml` (any casing, by filename ‚Äî reuse [BmadArtifactAdapter](../../src/SpecScribe/BmadArtifactAdapter.cs)'s `sprint-status.yaml` convention, [line 163](../../src/SpecScribe/BmadArtifactAdapter.cs), don't hardcode a second literal) and `_bmad/config.toml`. **Check `IsDataSource` before `IsEpicsRelated`** in `CreateTimer` so sprint-status.yaml takes the data route, not the epics route.
- **The route re-parses the data it feeds and rewrites those surfaces.** The correct, simplest implementation is a **full `GenerateAll()`** (re-parses `_sprint`, re-reads module/retro/epics, rewrites the dashboard + sprint page + everything). ADR 0005 ¬ß3's scoped-re-render is the perf follow-up (R6.4, **OUT** ‚Äî deferred), not correctness. A dashboard/sprint-scoped variant (re-parse only sprint + `WriteSprint` + `WriteIndex`) is a valid optimization **if** you can re-run the same sprint ingestion `GenerateAll` uses ([SiteGenerator.cs:87‚Äì96](../../src/SpecScribe/SiteGenerator.cs)) without side effects ‚Äî but **prefer the full pass** for correctness and to match the extension's per-refresh behavior; note the perf tradeoff and leave scoping to R6.4. Name it e.g. `RegenerateFromDataSource(fullPath)` returning a `GenerationEvent` (Updated), so the watch footer logs it like the others.
- **Extension side needs none of this dispatch logic** ‚Äî each extension refresh already spawns a fresh `specscribe webview` ‚Üí `GenerateAll()`. The extension fix for AC #1 is **purely** broadening the watch globs (fact #7). That asymmetry (extension = glob only; core = glob + re-guard + a new route) is exactly what the epic AC means by "fixed in both layers ‚Ä¶ the core side is more than the `Filter` property."

### The config.toml asymmetry ‚Äî be explicit, don't fake it (fact #5)

- **Extension:** fully fixed. New `_bmad/config.toml` watch glob ‚Üí fresh spawn ‚Üí `ForgeOptions.Resolve()` re-reads `project_name` ‚Üí new `SiteTitle` throughout. Nothing more needed.
- **Core `watch` mode:** the long-lived generator's `_options.SiteTitle` is fixed at resolve time. The new data route can **re-render** on a config.toml change, but the rendered pages keep the **old** project name until the process restarts, unless the route **re-reads** `project_name` and updates the branding. **Decision:** for this story, wire config.toml into the **watch trigger** (so the site re-renders ‚Äî covering any *other* config-derived behavior and keeping parity with the extension's data-source list), and **explicitly document** that a project-name change in core `watch` mode needs a restart to re-brand (the SiteTitle is baked into `_options`). Re-resolving `ForgeOptions` mid-watch is a larger change (it would rebuild the generator) ‚Äî call it out as a deferred-work item rather than half-implementing it. The extension (the actual "panel stays live" surface this AC is about) has no such limit, so the user-facing promise holds where it matters.

### AC #2a: resolved watch roots on the payload (R6.2) + the uniform subdir fix (fact #8, #9)

Add to the `webview` payload ‚Äî mirroring `configuredOutputRoot` exactly ([Commands.cs:103‚Äì108](../../src/SpecScribe/Commands.cs)):

| `adrRoot` | `Path.GetRelativePath(RepoRoot, AdrSourceRoot).Replace('\\','/')` | watcher base for `docs/adrs` (default) or a custom ADR root |
| `repoRoot` | workspace-relative offset from `folder` to `RepoRoot` (e.g. `""`/`"."` at root; `"../.."` when opened two levels deep) | so the shim resolves the **absolute** repo root once and anchors watchers **and** 6.10 reveal-source to it |

- **All three are pure, side-effect-free, unit-testable** (like `ResolveConfiguredOutputRoot`) ‚Äî add them to `Commands.cs` as static resolvers and cover with C# tests. Put them on `WebviewBundle` **or** compute them in `SerializePayload` from the pre-redirect `resolved` options (like `configuredOutputRoot`, which is passed in from `resolved`, **not** the scratch-redirected `options`). **Use the pre-redirect `resolved`** so the roots are the project's real roots, never the temp scratch dir.
- **Extension:** add `sourceRoot?`/`adrRoot?`/`repoRoot?` to the `WebviewPayload` interface ([extension.ts:75‚Äì86](../../extension/src/extension.ts)), all optional so an older core still parses. In `SpecScribeStore`, resolve `const repoAbs = path.resolve(this.folder.uri.fsPath, payload.repoRoot ?? '.')` and build watchers as `new vscode.RelativePattern(vscode.Uri.file(repoAbs), \`${payload.sourceRoot}/**/*.{md,yaml,yml}\`)` and one for `adrRoot` + one for `_bmad/config.toml`. **Fall back to today's literal globs** (`_bmad-output`, `docs/adrs`) when the payload lacks the roots (older core / not-yet-loaded), so nothing regresses.
- **Chicken-and-egg (watchers need a payload, the payload needs a spawn):** today `startWatching()` runs at bind time, **before** any load. Restructure so the watchers are **(re)built from the resolved roots once the first payload lands** ‚Äî e.g. build a **bootstrap** watcher set on the literal fallback globs at bind time (so an edit before first open still triggers a lazy load), then **rebuild** them from the payload roots on the first successful `load()` (dispose the bootstrap set first). Keep the `if (this.cache)` reload guard. Do this inside the store; don't leak watcher lifecycle into the panel.
- **Harmonize 6.10 reveal-source (fact #9):** 6.10 resolves `path.join(cwd, sourcePath)` with `cwd = folder.uri.fsPath`. Change it (and 6.9's tree "Open Source" if 6.10 didn't already unify it ‚Äî 6.10 Task 4) to join against the **resolved repo root** (`repoAbs`), so reveal-source is correct on a subdir-open too. **One convention:** core emits repo-relative paths + a `repoRoot` offset; the shim resolves the repo root once and joins everything to it. Remove any remaining `_bmad-output` literal from the reveal path.

### AC #2b: visibility-aware refresh across the shared store (R6.3, fact #10)

## Dev Notes

### The core defect is a mis-route, not just a filter (facts #1‚Äì#4)
The obvious read of R6.1 ‚Äî "widen `Filter = "*.md"`" ‚Äî is a **trap** that leaves the bug half-fixed. `sprint-status.yaml` lives under `implementation-artifacts/`, so once it gets past the filter it hits `IsEpicsRelated ‚Üí RegenerateEpics`, which **deliberately never re-parses sprint state** (AD-5 watch parity). That's why the epic AC says "the core side is more than the `Filter` property."

### The extension is a fresh spawn every time ‚Äî the config.toml half is easy there, hard in core (fact #5)
Each `specscribe webview` spawn re-runs `ForgeOptions.Resolve()`, so a `config.toml` project-name change is picked up **for free** in the extension once the glob fires. The long-lived core `watch` generator holds a fixed `_options.SiteTitle` and **cannot** re-brand mid-run without rebuilding the generator. Wire config.toml into the watch trigger (re-render), but **document honestly** that core `watch` needs a restart to change the displayed project name ‚Äî don't fake it. The "panel stays live" promise (Story 6.4 AC #3) is about the **extension**, where it fully holds.

### The payload `repoRoot` datum is finally real ‚Äî 6.10 punted it here on purpose (facts #8, #9)
