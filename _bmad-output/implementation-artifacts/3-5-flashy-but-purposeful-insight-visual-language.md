# Story 3.5: Flashy but Purposeful Insight Visual Language

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a stakeholder consuming status quickly,
I want insight visuals to feel impressive but still actionable,
so that demos and day-to-day usage both benefit from clarity and impact.

## Acceptance Criteria

1. **Given** insight modules render charts and drill paths **When** the page loads and interactions occur **Then** transitions communicate state changes clearly **And** motion remains bounded, meaningful, and performance-safe. [Source: epics.md#Story 3.5, lines 479-491; UX-DR20; DESIGN.md "Motion philosophy"]
2. **Given** accessibility constraints apply **When** flashy visual affordances are enabled **Then** equivalent text and non-color cues remain present **And** reduced-motion settings preserve full informational meaning. [Source: epics.md#Story 3.5; UX-DR18; UX-DR8; UX-DR17; NFR6]

## Dev Notes

- **This is the capstone motion/visual-language story of Epic 3 — it harmonizes, it does not invent surfaces.** Stories 3.1 (git pulse), 3.3 (coverage), 3.4 (tree), and 3.6 (funnel) each *deliberately shipped static* and wrote "motion is Story 3.5's job" into their own task lists [Source: [3-3…:57](../../_bmad-output/implementation-artifacts/3-3-agent-and-workflow-structure-coverage-insights.md), [3-4…:33](../../_bmad-output/implementation-artifacts/3-4-interactive-tree-views-for-project-and-artifact-structure.md)]. Story 3.5's deliverable is a single, tokenized, reduced-motion-safe motion vocabulary applied consistently across all insight surfaces so the portal feels like one deliberate product — impressive on load/demo, bounded and meaningful in daily use. It is NOT a mandate to add new charts or a JS drill engine.
- **⚠️ Sequencing — implement 3.5 LAST in Epic 3.** At authoring time, siblings 3.1–3.4 and 3.6 are all `ready-for-dev` (not `done`), so several surfaces this story animates (coverage panel, tree, funnel, git-pulse panels) **do not exist in the codebase yet** — only the Story 1.5 dashboard charts + the commit heatmap are built. Best path: land 3.1–3.4 and 3.6 first, then this story. If 3.5 is implemented before a sibling lands, write the motion rules **defensively** — key them to shared/opt-in selectors so an absent surface is a harmless no-op, animate only what actually renders (heatmap, sunburst, donuts, progress bars, whichever Epic 3 panels exist), and record any surface-specific motion left for later in Completion Notes. Never fabricate a sibling's markup to animate it. *(This sequencing is flagged to the human at story-creation; confirm ordering before dev.)*
- **The motion contract is TWO paired media blocks — extend them, never rewrite.** Story 1.5 built the `no-preference` entrance block ([:2135](../../src/SpecScribe/assets/specscribe.css)) and left it side-by-side with Story 1.4's `reduce` block ([:2156](../../src/SpecScribe/assets/specscribe.css)) *specifically so this story could extend the pair*. Everything you add hangs off one of those two blocks. The `reduce` half is a hard accessibility invariant (AC #2, UX-DR18/8): every animation you add on the `no-preference` side must be neutralized on the `reduce` side with zero information loss.
- **One motion vocabulary, mirroring the status-token discipline.** The whole point is that "flashy" stops being ad-hoc: today timings are scattered literals (0.9s, 0.6s, 0.12s, 0.15s, 0.3s). Tokenize them (`--motion-*`) so every insight surface animates with the same feel, exactly as `--status-*` made one color-per-stage the single source ([specscribe-status-token-system], [StatusStyles.cs](../../src/SpecScribe/StatusStyles.cs)). Motion tokens are a NEW parallel family — do not reshape or overload the status tokens.
- **Charts are pure SVG + CSS, no JS — and that stays true here.** The deliberate project stance is no client-side chart engine; the ONE sanctioned exception is the small `specscribe.js` (tooltips + copy + menu dismissal) [Source: [charting-is-pure-svg-no-js], [Charts.cs:6-8](../../src/SpecScribe/Charts.cs)]. Prefer pure CSS for all of this story's motion (entrances are `@keyframes` + `animation`; legend emphasis wants scoped `:has()`/sibling selectors). Only if legend emphasis genuinely can't be expressed in CSS may you add a **minimal class-toggle** to the existing `specscribe.js` — no new file, no library, no drill/zoom/hash engine, no dark-mode/theme system (all permanently out of scope per the Story 1.5 guardrails).
- **Visual-polish timings are seed-level; accessibility is not.** The architecture spine is explicit: "Exact visual polish values (timings, shadows, micro-motion tuning) are seed-level as long as shared semantics and accessibility invariants remain intact" [Source: [ARCHITECTURE-SPINE.md#Seed, Not Invariant:103](../../_bmad-output/specs/spec-specscribe/ARCHITECTURE-SPINE.md)]. So you have latitude to tune durations/easings to feel right — but the reduced-motion parity, status-never-color-only, and keyboard-reachability invariants are non-negotiable.
- **Insight motion is additive and never owns baseline success.** Per AD-4, insight providers/surfaces enrich output but must never block generation [Source: [ARCHITECTURE-SPINE.md#AD-4:58](../../_bmad-output/specs/spec-specscribe/ARCHITECTURE-SPINE.md)]. This is a CSS/asset story, so the risk is low, but keep it true: no motion work should introduce a generation-time failure path, and every surface must render correctly with CSS/JS stripped.
- **Reuse, don't reinvent, the existing hover grammar.** Story 1.5 D1/D3 already defined the interaction grammar (interactive → lift+accent+shadow; data-hover → highlight+tooltip; static → nothing) and the sunburst hover-invert ([specscribe.css:1118](../../src/SpecScribe/assets/specscribe.css)). Legend emphasis (Task 3) must read as the same language — dim the non-matching, emphasize the match — not a novel effect.
- **`docs/live` / generated-output caveat.** Edit only the embedded `src/SpecScribe/assets/specscribe.css` and (if unavoidable) `src/SpecScribe/assets/specscribe.js`; the `docs/live/*` copies are vestigial/gitignored and stale, and `StylesheetTests` loads from the assembly manifest, not disk [Source: [generate-output-dir-is-specscribeoutput]]. The default generate output dir is `SpecScribeOutput`. Remember the Story 1.5 review lesson: asset hrefs are cache-busted by build version ([PathUtil.cs](../../src/SpecScribe/PathUtil.cs) `RenderHeadOpen`), so a CSS/JS change ships without a stale-cache artifact — do not reintroduce un-versioned hrefs.
- **No `architecture.md` exists** — the governing docs are the spec kernel ([SPEC.md](../../_bmad-output/specs/spec-specscribe/SPEC.md), [ARCHITECTURE-SPINE.md](../../_bmad-output/specs/spec-specscribe/ARCHITECTURE-SPINE.md)) plus the UX design ([DESIGN.md](../../_bmad-output/planning-artifacts/ux-designs/ux-SpecScribe-2026-07-05/DESIGN.md), [EXPERIENCE.md](../../_bmad-output/planning-artifacts/ux-designs/ux-SpecScribe-2026-07-05/EXPERIENCE.md)).
